# <a name="start-a-debugging-session-for-a-uwp-app"></a>Запуск сеанса отладки для приложения UWP

В этой статье описывается запуск сеанса отладки Visual Studio для приложений универсальной платформы Windows (UWP). Приложения универсальной платформы Windows, которые могут быть написаны в XAML и C++, XAML и C#/Visual Basic. Чтобы начать отладку приложения универсальной платформы Windows, настроить сеанс отладки и Выбор способа запуска приложения.

::: moniker range=">=vs-2019"
> [!NOTE]
> Начиная с Visual Studio 2019 г uwp для HTML и JavaScript больше не поддерживаются.
::: moniker-end
::: moniker range="vs-2017"
В Visual Studio 2017 большинство команд и параметров, описанных в этой статье также применимы для приложений универсальной платформы Windows для HTML и JavaScript. Где команды различаются для управляемых и C++ приложений, приложений JavaScript обычно такие же, как команды для C++ приложений универсальной платформы Windows.
::: moniker-end

##  <a name="BKMK_The_easy_way_to_start_debugging"></a>Запуск отладки из панели инструментов Visual Studio

Самый простой способ настройки и запуска отладки — на стандартной панели инструментов Visual Studio.

![Отладка с помощью панели инструментов](../debugger/media/vsrun_select_target_device.png)

1. Из **конфигурации** раскрывающегося списка на **стандартный** панели инструментов, выберите **Отладка**.

1. Из **платформы** раскрывающемся списке выберите целевую платформу для построения.

1. Из раскрывающегося списка рядом с зеленой стрелкой выберите целевой объект отладки. Можно выбрать локальный компьютер, устройства, подключенного к direct, локального симулятора Visual Studio, удаленное устройство или эмулятор.

1. Чтобы начать отладку, щелкните зеленый **запустить** стрелку на панели инструментов или выберите **Отладка** > **начать отладку**, или нажмите клавишу **F5**.

   Visual Studio создает и запускает приложение с прикрепленным отладчиком.

Отладка продолжается, пока не будет достигнута точка останова, приостановления выполнения вручную, необработанного исключения или завершения приложения.

###  <a name="BKMK_Choose_the_deployment_target"></a> Параметры целевой объект развертывания

Вы можете задать целевой объект отладки на панели инструментов Visual Studio или страницу свойств отладки проекта. Выберите один из следующих вариантов.

|||
|-|-|
|**Локальный компьютер**|Отладка приложения в текущем сеансе на локальном компьютере.|
|**Имитатор**|Отладка приложения в симуляторе Visual Studio для приложений универсальной платформы Windows. Имитатор является окно рабочего стола, которое имитирует устройство функции, наподобие touch жестов и поворот устройства, который не существует на локальном компьютере. Симулятор доступен только если приложения **Минимальная версия целевой. Версия** меньше или равно операционной системы на локальном компьютере. Дополнительные сведения см. в разделе [запуску приложений UWP в симуляторе](../debugger/run-windows-store-apps-in-the-simulator.md).|
|**Удаленный компьютер**|Отладка приложения на устройстве, подключенном к локальному компьютеру по сети или кабеля Ethernet. Инструменты удаленной отладки для Visual Studio должны быть установлены и запущены на удаленном устройстве. Дополнительные сведения см. в разделе [запуску приложений UWP на удаленном компьютере](../debugger/run-windows-store-apps-on-a-remote-machine.md).|
|**Устройство**|Отладка приложения на устройстве, подключаемом к USB. Это должно быть разблокировано разработчика и разблокировать текста на экране.|
|**Эмулятор мобильных устройств**|Загрузите эмулятор, указанный в имени эмулятор, развернуть приложение и начать отладку. Эмуляторы предоставляются только на компьютерах с поддержкой Hyper-V.|

##  <a name="BKMK_Open_the_debugging_property_page_for_the_project"></a> Настройка отладки на странице свойств проекта

Чтобы настроить дополнительные параметры отладки, используйте страницу свойств отладки проекта.

**Чтобы открыть свойства отладки:**

1. В **обозревателе решений**, выберите проект, а затем выберите **свойства** значок, или щелкните правой кнопкой мыши проект и выберите **свойства**.

1. В левой части **свойства** области:

   - Для C# и приложений Visual Basic, выберите **Отладка**.

     ![C#и страницу свойств отладки проекта Visual Basic](../debugger/media/dbg_csvb_debugpropertypage.png)

   - Для C++ приложений, выберите **свойства конфигурации** > **Отладка**.

     ![C++Страницу свойств отладки приложения UWP](../debugger/media/dbg_cpp_debugpropertypage.png)

###  <a name="BKMK_Choose_the_debugger_to_use"></a> Выбор отладчика

Для C# и приложений Visual Basic, Visual Studio отлаживает управляемого кода по умолчанию. Вы можете выполнять отладку кода других или дополнительных типов. Можно также задать **тип отладчика** значения для все фоновые задачи, которые являются частью проекта.

В приложениях C++ Visual Studio отлаживает машинный код по умолчанию. Вы можете выполнить отладку конкретных типов кода вместо или в дополнение к машинного кода.

**Чтобы указать типы кода для отладки:**

- Для C# и приложения Visual Basic, выберите один из следующих отладчиков из **тип приложения** и **тип фонового процесса** раскрывающиеся списки в разделе **тип отладчика** на **Отладка** страницу свойств.

- Для C++ приложения, выберите один из следующих отладчиков из **тип отладчика** раскрывающегося списка на **Отладка** страницу свойств.

|||
|-|-|
|**Только управляемый код**|Отладка управляемого кода в приложении. Код JavaScript и машинный код C/C++ игнорируются.|
|**Только машинный код**|Отладка машинного кода C/C++ в приложении. Управляемый код и код JavaScript игнорируются.|
|**Смешанный (управляемый и машинный)**|Отладка машинного кода C/C++ и управляемого кода в приложении. Код JavaScript игнорируется. В C++ проектов, этот параметр называется **управляемый и машинный**.|
|**Сценарий**|Отладка кода JavaScript в приложении. Управляемый код и машинный код игнорируются.|
|**Машинный код со скриптом**|Отладка машинного кода C/C++ и кода JavaScript в приложении. Управляемый код учитывается. В C++ проектов или фона только задачи.|
|**Только GPU (C++ AMP)**|Отладка машинного кода C++, который выполняется в графическом процессоре (GPU). Доступные только для проектов C++.|

###  <a name="BKMK__Optional__Disable_network_loopbacks"></a> Отключить сетевое замыкание на себя (необязательно)

 Для обеспечения безопасности приложения универсальной платформы Windows, которая устанавливается с стандартным образом не может выполнять сетевые вызовы к устройству, на котором он установлен на. Visual Studio среди развернутые приложения из этого правила по умолчанию, чтобы вы могли тестировать процедуры обмена данными на одном компьютере. До выпуска приложения, необходимо протестировать приложение без этого исключения.

**Удаление исключения из правила сетевого замыкания на себя:**

-   Для C# и приложений Visual Basic снимите флажок **разрешить замыкание на себя в локальной сети** флажок **параметры запуска** на **Отладка** страницу свойств.

-   Для визуального C++ приложений, выберите **нет** из **разрешает локальном сетевое замыкание на себя** раскрывающегося списка на **Отладка** страницу свойств.

###  <a name="BKMK__Optional__Reinstall_the_app_when_you_start_debugging"></a> Переустановите приложение при запуске отладки (необязательно)
 Для диагностики проблем с установкой C# или приложения Visual Basic, выберите **удалить и переустановить Мой пакет** на **Отладка** страницу свойств. Этот параметр восстанавливает исходную установку при запуске отладки. Этот параметр недоступен для C++ проектов.

###  <a name="BKMK__Optional__Disable_authentication_requirement_to_start_the_remote_debugger"></a> Настройка параметров проверки подлинности для удаленной отладки

По умолчанию, необходимо указать учетные данные Windows для запуска удаленного отладчика, при выборе **удаленный компьютер** как целевой объект развертывания. Вы можете изменить требование проверки подлинности.

**Универсальный (незашифрованный протокол)** используется проверка подлинности для устройств Интернета вещей, Xbox и HoloLens и создателя обновления или более поздней версии Windows 10.

**Чтобы изменить метод проверки подлинности:**

- Для C# и приложений Visual Basic на **Отладка** страницу свойств, выберите **удаленный компьютер** как **целевое устройство**. Выберите **None** или **универсальный (незашифрованный протокол)** для **режим проверки подлинности**.

- Для C++ приложений, выберите **удаленный компьютер** под **отладчик для запуска** на **Отладка** страницу свойств. Выберите **без проверки подлинности** или **универсальный (незашифрованный протокол)** для **тип проверки подлинности**.

> [!CAUTION]
> Не имеет безопасности сети, при запуске удаленного отладчика в **None** или **универсальный (незашифрованный протокол)** режимы. Выберите этих режимов только в доверенных сетях, в которых вы являетесь Конечно, не подвержены вредоносный код или опасного трафика.

##  <a name="BKMK_Start_the_debugging_session"></a> Параметры запуска отладки

При выборе **Отладка** > **начать отладку** или нажмите клавишу **F5**, Visual Studio запускает приложение с подключенным отладчиком. Выполнение продолжается до достижения точки останова, приостановления выполнения вручную, необработанного исключения или завершения приложения.

###  <a name="BKMK_Start_debugging__F5__but_delay_the_app_start"></a> Начать отладку, но задержка запуска приложения

По умолчанию Visual Studio запускает приложение немедленно в том случае, когда вы запускаете отладку. Можно также задать приложение для запуска в режиме отладки, но запуск приложения вне отладчика. Например, может потребоваться отладка запуск приложений из Windows **запустить** меню или отладки фонового процесса в приложении. Если выбран этот параметр, приложение запускается в отладчике при запуске.

**Отключение запуска автоматического приложения:**

- Для C# и приложений Visual Basic, выберите **не запускать, а отлаживать мой код при открытии** под **параметры запуска** на **Отладка** страницу свойств.

- Для C++ приложений, выберите **нет** из **запустить приложение** раскрывающегося списка на **Отладка** страницу свойств.

Дополнительные сведения об отладке фоновых задач см. в разделе [триггера приостановки, возобновления и фоновых событий для приложений UWP](../debugger/how-to-trigger-suspend-resume-and-background-events-for-windows-store-apps-in-visual-studio.md).

###  <a name="BKMK_Start_an_installed_app_in_the_debugger"></a> Отладка приложения универсальной платформы Windows установлен или не запущен

Можно использовать **отлаживать установленный пакет приложения** для отладки приложения UWP, которое уже установлен или не запущен на локальном или удаленном устройстве. Приложение могло быть установлено из Microsoft Store, или может быть проект Visual Studio. Например приложение может иметь специальная система построений, не использующей Visual Studio.

Установленное приложение можно запустить немедленно, или можно разместить его для выполнения в отладчике при запуске с помощью другого метода. Дополнительные сведения см. в разделе [триггера приостановки, возобновления и фоновых событий для приложений UWP)](../debugger/how-to-trigger-suspend-resume-and-background-events-for-windows-store-apps-in-visual-studio.md).

Чтобы запустить приложение универсальной платформы Windows установлен или не запущен в отладчике, выберите **Отладка** > **другие целевые объекты отладки** > **отлаживать установленный пакет приложения**. Дополнительные инструкции см. в разделе [отладка установленного пакета приложения](../debugger/debug-installed-app-package.md).

###  <a name="BKMK_Attach_the_debugger_to_a_running_app_"></a> Подключение отладчика к выполняемому приложению Windows 8.x

Чтобы подключить отладчик к приложению [!INCLUDE[win8_appname_long](../debugger/includes/win8_appname_long_md.md)] , нужно воспользоваться диспетчером отлаживаемых пакетов и настроить приложение на выполнение в режиме отладки. Диспетчер отлаживаемых пакетов устанавливается вместе с инструментами удаленной отладки для Visual Studio.

1. Установка инструментов удаленной отладки для Visual Studio на устройстве, где установлено приложение. Дополнительные сведения см. в разделе [установки средств удаленного](../debugger/remote-debugging.md).

1. В Windows **запустить** экране, найдите и запустите **диспетчер отлаживаемых пакетов**.

   Отобразится окно PowerShell, настроенное для работы с командлетами AppxDebug.

1. Укажите идентификатор "полноеимяпакета" приложения.

   1. Чтобы просмотреть список, содержащий полное имя пакета для всех приложений, введите `Get-AppxPackage` командной строке PowerShell.

   1. В окне PowerShell введите `Enable-AppxDebug <PackageFullName>`, где \<"полноеимяпакета" > — идентификатор "полноеимяпакета" данного приложения.

1. Выберите **Отладка** > **Присоединение к процессу**.

1. В **присоединение к процессу** диалоговом окне укажите это удаленное устройство в **цель подключения** поле.

   Можно ввести имя устройства, выберите его из раскрывающегося списка в **цель подключения** , или **найти** найти устройства в **удаленные подключения** диалоговое окно.

1. Чтобы указать тип кода, необходимо выполнить отладку, рядом с полем **присоединить к** выберите **выберите**.

1. В **Выбор типа кода** диалогового окна выберите либо:
   - **Автоматически определять тип отлаживаемого кода**, или
   - **Отладить код следующих типов**, а затем выберите один или несколько типов кода из списка.

1. В **доступные процессы** выберите процесс приложения для отладки.

1. Выберите **присоединить**.

 Visual Studio подключает отладчик к процессу. Выполнение продолжается до достижения точки останова, приостановления выполнения вручную, необработанного исключения или завершения приложения.

::: moniker range="vs-2017"
> [!NOTE]
> Приложения JavaScript выполняются в экземпляре процесса *wwahost.exe*. Если выполняется более одного приложения JavaScript, необходимо знать числовой идентификатор процесса (PID) вашего приложения *wwahost.exe* процесс для присоединения к ней.
>
> Самый простой способ подключить к приложению JavaScript заключается в закрытии всех других приложений JavaScript. Или можно обратить внимание идентификаторов процесса выполнения *wwahost.exe* процессы в Windows диспетчер задач перед запуском приложения. При запуске приложения, его *wwahost.exe* PID будет тот, который отличается от тех, которые вы записали ранее.
::: moniker-end
