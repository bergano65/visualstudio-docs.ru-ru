# <a name="error-unable-to-start-debugging-on-the-web-server"></a>Ошибка. Не удается запустить отладку на веб-сервере

При попытке отладить приложение ASP.NET на веб-сервере, может появиться это сообщение об ошибке: `Unable to start debugging on the Web server`.

Часто эта ошибка возникает, так как произошла ошибка или изменения конфигурации, необходимо обновить пулы приложений и сброс IIS. Вы можете сбросить IIS, откройте командную строку и введите `iisreset`.

## <a name="specificerrors"></a>Что такое подробное сообщение об ошибке?

`Unable to start debugging on the Web server` Сообщения является универсальным. Как правило более конкретное сообщение включается в строку ошибки и это может помочь определить причину проблемы или поиск для получения более точных исправления. ## <a name="IISlist"></a> IIS не содержит веб-сайт, соответствующий запуск URL-адрес

- Перезапустите Visual Studio с правами администратора и повторите отладки. (В некоторых сценариях отладки ASP.NET требует повышенных привилегий).

    Можно настроить Visual Studio, чтобы всегда выполнять от имени администратора, щелкните правой кнопкой мыши значок ярлыка Visual Studio, выбрав **свойства > Дополнительно**и затем решили всегда выполнять от имени администратора.

## <a name="web_server_config"></a> Веб-сервер настроен неправильно

- См. в разделе [ошибка: веб-сервер настроен правильно](../debugger/error-the-web-server-is-not-configured-correctly.md).

## <a name="unabletoconnect"></a> Не удалось подключиться к веб-сервера

- Запуск Visual Studio и веб-сервера на одном компьютере и отладке с помощью **F5** (а не **присоединение к процессу**)? Откройте свойства проекта и убедитесь, что проект настроен для подключения к правильный веб-сервер и запустите URL-адрес. (Откройте **свойства > Web > серверы** или **свойства > Отладка** в зависимости от типа проекта. Для проекта веб-форм, откройте **страницы свойств > Параметры запуска > Server**.)

- В противном случае перезапуск пула приложений и затем перезапустите службы IIS. Дополнительные сведения см. в разделе [Проверьте конфигурацию IIS](#vxtbshttpservererrorsthingstocheck).

## <a name="webservertimeout"></a> Веб-сервер не ответил вовремя

- Перезапустите службы IIS и повторите попытку отладки. Может быть подключено несколько экземпляров отладчик к процессу IIS; Сброс завершает их. Дополнительные сведения см. в разделе [Проверьте конфигурацию IIS](#vxtbshttpservererrorsthingstocheck).

## <a name="msvsmon"></a> Вероятно, монитор удаленной отладки Microsoft Visual Studio (msvsmon.exe) не запущен на удаленном компьютере

- При отладке на удаленном компьютере, убедитесь, что у вас есть [установлены и работают удаленный отладчик](../debugger/remote-debugging.md). Если в сообщении упоминаются брандмауэр, убедитесь, что [исправьте портов в брандмауэре](../debugger/remote-debugger-port-assignments.md) открыты, особенно в том случае, если вы используете брандмауэр третьих лиц.
- Если вы используете файл HOSTS, убедитесь, что он правильно настроен. Например, при отладке с помощью **F5** (вместо **присоединение к процессу**), должно включать один и тот же адрес проекта, как и в свойствах проекта, файл HOSTS **свойства > Web > серверы**  или **свойства > Отладка**, в зависимости от типа проекта.

## <a name="server_error"></a> Удаленный сервер вернул ошибку

Проверьте ваш [файл журнала IIS](https://support.microsoft.com/help/943891/the-http-status-code-in-iis-7-0--iis-7-5--and-iis-8-0) дополнительные коды ошибок и Дополнительные сведения и этот IIS 7 [блога](https://blogs.iis.net/tomkmvp/troubleshoot-a-403).

Кроме того ниже приведены некоторые распространенные коды ошибок и несколько предложений.
- (403) Запрещено. Существует несколько возможных причин этой ошибки, поэтому файл журнала и параметрами безопасности IIS для веб-сайта. Убедитесь, что файл web.config сервера включает `debug=true` в элементе compilation. Убедитесь, что в папке веб-приложения есть нужные разрешения и что конфигурацию пула приложений верно (изменился пароль). См. в разделе [Проверьте конфигурацию IIS](#vxtbshttpservererrorsthingstocheck). Если параметры уже заданы правильно, и при локальной отладке, также убедитесь, что вы подключаетесь к правильный тип сервера и URL-адрес (в **свойства > Web > серверы** или **свойства > Отладка**, в зависимости от типа проекта).
- (503) Сервер недоступен. Пул приложений была остановлена из-за ошибки или изменения конфигурации. Перезапуск пула приложений.
- (404) Не найдено. Убедитесь, что пул приложений настроен для правильной версии ASP.NET.

## <a name="aspnet"></a> Не удается запустить отладку ASP.NET

- Перезапуск пула приложений и перезапустите службы IIS. Дополнительные сведения см. в разделе [Проверьте конфигурацию IIS](#vxtbshttpservererrorsthingstocheck).
- Если вы выполняете операции перезаписи URL-адрес, протестируйте базовый файл web.config с без перезаписи URL-адрес. См. в разделе **Примечание** о URL-адрес перезаписи модуля в [Проверьте конфигурацию IIS](#vxtbshttpservererrorsthingstocheck).

## <a name="cannot_connect"></a> Отладчик не может подключиться к удаленному компьютеру

Если вы производите отладку локально, эта ошибка может возникать, так как Visual Studio является 32-разрядное приложение, он использует 64-разрядной версии удаленного отладчика для отладки 64-разрядных приложений. Откройте свойства проекта и убедитесь, что проект настроен для подключения к правильный веб-сервера и URL-адрес. (Откройте **свойства > Web > серверы** или **свойства > Отладка** в зависимости от типа проекта.)

Кроме того Если вы используете файл HOSTS, убедитесь, что он правильно настроен. Например, файл HOSTS, должно включать один и тот же адрес проекта, как и в свойствах проекта **свойства > веб > серверов** или **свойства > Отладка**, в зависимости от типа проекта.

## <a name="see_help"></a> Сведения об общих ошибках конфигурации см. в справке. Под управлением веб-страницы вне отладчика может предоставить дополнительные сведения.

- Вы используете Visual Studio и веб-сервера на одном компьютере? Откройте свойства проекта и убедитесь, что проект настроен для подключения к правильный веб-сервер и запустите URL-адрес. (Откройте **свойства > Web > серверы** или **свойства > Отладка** в зависимости от типа проекта.)

- Если это не поможет, или при удаленной отладке, выполните шаги [Проверьте конфигурацию IIS](#vxtbshttpservererrorsthingstocheck).

##  <a name="vxtbshttpservererrorsthingstocheck"></a> Проверьте конфигурацию IIS

После выполнения действия, описанные здесь, чтобы устранить проблему и перед повторной попыткой для отладки также может потребоваться сброс IIS. Это можно сделать, открыв командную строку и введя `iisreset`.

* Остановка и перезапуск пулов приложений IIS, а затем повторите попытку.

    Пул приложений была остановлена из-за ошибки. Или другой внесенные изменения конфигурации требуется остановка и перезапуск пула приложений.

    > [!NOTE]
    > Если отслеживает остановки пула приложений, может потребоваться удалить модуль переопределения URL-адрес с помощью панели управления. Вы можете повторно установить его с помощью установщика веб-платформы (WebPI). Эта проблема может возникать после обновления значительного объема системной.

* Проверьте конфигурацию пула приложений, его при необходимости исправьте и повторите попытку.

    Пул приложений могут быть настроены для версии ASP.NET, соответствует ли проект Visual Studio. Обновление версии ASP.NET в пуле приложений и перезапустите его. Подробные сведения см. в разделе [IIS 8.0 с ASP.NET 3.5 и ASP.NET 4.5](/iis/get-started/whats-new-in-iis-8/iis-80-using-aspnet-35-and-aspnet-45).

    Кроме того Если пароль учетные данные изменились, может потребоваться обновить их в пул приложений или веб-сайта.  В пуле приложений, обновите учетные данные в **Дополнительные параметры > модель процесса > удостоверение**. Для веб-сайта, обновите учетные данные в **основные параметры > подключиться как...** . Перезапуск пула приложений.

* Убедитесь, что папку веб-приложение имеет необходимые разрешения.

    Убедитесь, что вы предоставляете IIS_IUSRS, IUSR, или конкретного пользователя, связанное с [пул приложений](/iis/manage/configuring-security/application-pool-identities) чтение и выполнение права для папки веб-приложения. Устраните проблему и перезапустите пул приложений.

* Убедитесь, что необходимая версия ASP.NET установлен на сервере IIS.

    Несовпадение версий ASP.NET в IIS и в своем проекте Visual Studio может вызвать эту проблему. Может потребоваться задать версию платформы в файле web.config. Чтобы установить ASP.NET на IIS, используйте [установщика веб-платформы (WebPI)](https://www.microsoft.com/web/downloads/platform.aspx). Кроме того, см. в разделе [IIS 8.0 с ASP.NET 3.5 и ASP.NET 4.5](/iis/get-started/whats-new-in-iis-8/iis-80-using-aspnet-35-and-aspnet-45) или для ASP.NET Core, [узла в Windows со службами IIS](https://docs.asp.net/en/latest/publishing/iis.html).

* Устраните ошибки проверки подлинности, если используется только IP-адрес

     По умолчанию предполагается, что IP-адреса являются частью Интернет-зоны, и проверка подлинности NTLM для них не выполняется. Если веб-сайт настроен в IIS, чтобы требовать проверку подлинности, проверка подлинности завершается ошибкой. Чтобы устранить эту проблему, можно указать имя удаленного компьютера, а не IP-адрес.

## <a name="other-causes"></a>Другие причины

Если конфигурации IIS не является причиной проблемы, попробуйте выполните следующие действия:

- Перезапустите Visual Studio с правами администратора и повторите попытку.

    Для Visual Studio, например с помощью веб-развертывания несколько сценариев отладки ASP.NET требует повышенных привилегий.

- Если выполняется несколько экземпляров Visual Studio, повторно откройте проект в одном экземпляре Visual Studio с правами администратора () и повторите попытку.

- Если вы используете файл HOSTS с локальных адресов, попробуйте использовать петлевой адрес вместо IP-адреса компьютера.

    Если вы не используете локальных адресов, убедитесь, что в файле УЗЛОВ имеется один и тот же адрес проекта, как и в свойствах проекта **свойства > Web > серверы** или **свойства > Отладка**, в зависимости от вашей Тип проекта.

## <a name="more-troubleshooting-steps"></a>Дополнительные действия по устранению неполадок

* Откройте страницу localhost в браузере на сервере.

     Если службы IIS не установлены правильно, должны отобразиться ошибки при вводе `http://localhost` в браузере.

     Дополнительные сведения о развертывании в службах IIS, см. в разделе [IIS 8.0 с ASP.NET 3.5 и ASP.NET 4.5](/iis/get-started/whats-new-in-iis-8/iis-80-using-aspnet-35-and-aspnet-45) и ASP.NET Core, [узла в Windows со службами IIS](https://docs.asp.net/en/latest/publishing/iis.html).

* Создать простое приложение ASP.NET на сервере (или воспользоваться файлом базовый файл web.config).

    Если не удается получить приложения для работы с отладчиком, попробуйте создать простое приложение ASP.NET локально на сервере и попробуйте отладить базовое приложение. (Может потребоваться использовать шаблон ASP.NET MVC по умолчанию.) Если базовое приложение можно отлаживать, который может помочь выявить отличия между двумя конфигурациями. Найдите различия в параметрах в файле web.config, такие как правила переопределения URL-адрес.
